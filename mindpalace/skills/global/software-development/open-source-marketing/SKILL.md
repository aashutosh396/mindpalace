# Open Source Marketing

Market OSS without being cringe. `Growth = (Real value) × (Discoverability) × (First-use experience)` — if any factor is zero, growth is zero.

What works: building in public, solving real problems, genuine community engagement, great docs/DX, celebrating contributors, consistent presence. What doesn't: spamming "check out my project", solutions seeking problems, transactional follows, "self-documenting code", taking sole credit, launch-and-disappear.

## GitHub optimization

**README is your landing page.** Structure: name + one-line description + badges + screenshot/GIF in action + "Why [project]?" (specific benefits) + quick start (<5 lines to value) + installation (all platforms, copy-paste) + usage examples + docs link + contributing + license.

README checklist: clear searchable name; one-liner ("A [type] for [audience] that [does what]"); badges (social proof/health); visual (GIF > screenshot > nothing); quick start <5 lines; differentiation; copy-paste install; real examples; docs link; contributing welcome.

Repo settings: keyword-rich description (≤100 chars), 5-10 topics, website link, semantic releases with changelogs, issue/PR templates, enable Discussions, enable Sponsors if wanted.

## Community building

Spaces: start with GitHub Discussions (low effort); add Discord at 50+ active users; Discourse forum for async/searchable at scale.

Principles: respond to issues within 48h (even "looking into it"), thank every contributor publicly, share roadmap and explain decisions, set response-SLA expectations, welcome newcomers with "good first issue" labels.

Contributor funnel: User → Star → Issue → PR → Regular → Maintainer. Optimize each transition (great README; welcoming templates; CONTRIBUTING.md; quick encouraging review; shared ownership).

## Contributor experience

CONTRIBUTING.md essentials: quick-start steps (fork/clone/install/branch/change/test/commit/push/PR), dev setup, code style, commit convention (Conventional Commits), PR process, "good first issue" pointer, where to ask.

Good first issues are genuinely approachable ("Add TypeScript types for X", "Fix typo", "Add test for Y") — not "refactor the codebase". For each: explain context/why, link relevant files, describe expected outcome, offer help.

## Launch

Pre-launch checklist: README polished, quick start works, docs exist, 3+ examples, tests pass, license, CONTRIBUTING.md, issue templates, social preview image, 5-10 topics.

Launch day: prep posts day before; HN post in the morning (≈6-8am PT, Tue-Thu); +1h Twitter thread; +2h relevant subreddits; respond to everything all day; thank everyone at end. Per platform: HN — descriptive title, motivation/tech in first comment, never ask for upvotes. Reddit — 2-3 relevant subs, read rules, be a member. Twitter — Problem→Solution→Demo→Link thread with GIF. Dev.to/Hashnode — "Why I built this" article.

Post-launch: week 1 fix bugs/respond; week 2 "what I learned" post; week 3 ship a feature; month 1 community/contributor docs; ongoing consistent releases.

## Sustainable growth

High-impact tactics: SEO-optimized docs, integration tutorials, comparison content ("X vs Y vs Z"), migration guides ("from Y to X"), "how we use X at [company]", performance benchmarks, conference talks.

Avoid burnout: response-SLA expectations, public roadmap/RFC for feature demands, recruit co-maintainers, scheduled office hours vs 24/7, code of conduct for negativity.

## Metrics

Value over vanity: active issues+PRs over stars; returning contributors over forks; weekly active users over downloads; community engagement over follower counts. Track via GitHub Insights/Traffic, Star History, npm-stat.
